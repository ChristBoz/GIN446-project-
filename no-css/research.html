Is HTML 5.2 the latest HTML version?

After HTML 5.2, HTML 5.3 was proposed but it was never finalized. The goal of HTML 5.3 was to introduce small enhancements and clarifications to the HTML5 specification. On December 14, 2017, the W3C released a working draft of HTML 5.3. The purpose of this draft was to improve upon current features and fix problems found in HTML 5.2 W3C. The Web Hypertext Application Technology Working Group (WHATWG) and the W3C agreed in May 2019 to stop developing distinct W3C versions of HTML, such as HTML 5.3 and DOM 4.1. Rather, they chose to concentrate on the WHATWG's "HTML Living Standard," which is updated and maintained by major browser vendors on a regular basis.

History of CSS

Håkon Wium Lie first proposed CSS (Cascading Style Sheets) in 1994, and he and Bert Bos worked together to develop it under the World Wide Web Consortium (W3C). Its goal was to enable people to distinguish between presentation (styles: colors, fonts, layout, etc.) and content (HTML), which HTML alone was unable to accomplish effectively.

December 1996 saw the release of CSS1, the first official version. Font styling, text properties, color, background, margins, padding, and the box model were among the new features it brought.

Then came CSS2, which was made available in May 1998 and included support for more sophisticated selectors, media types (screen, print), positioning (absolute, relative), and additional layout control. Later, CSS2.1 fixed some browser inconsistencies.

CSS3 was released in the middle of the 2000s; it was divided into modules rather than being a single, comprehensive specification, allowing for the independent development of various features. Animations, transitions, adaptable layouts, new selectors, and other features were all included in the modules. It facilitated faster innovation and increased CSS's flexibility.

More sophisticated CSS features have emerged in recent years, including grid layout, container queries, media queries, flexbox, responsive design, custom properties (variables), etc. It is constantly changing to assist developers in creating more maintainable and screen-size-adaptable designs.

CSS Rules and How They Work

A CSS rule, also known as a ruleset, is a declaration in CSS that instructs the browser on which HTML element or elements should have a particular appearance.

There are two primary components to a rule:

  • Selector(s)—which HTML element or elements to focus on
  • One or more declarations (property + value) inside { } in a declaration block

A declaration resembles property: value;

Here's the basic syntax:


selector {
  property1: value1;
  property2: value2;
  /* …more declarations… */
}

For example:


p {
  color: blue;
  font-size: 16px;
}

Since p is the selector, all paragraphs are targeted. The declarations color: blue; and font-size: 16px; form the full ruleset.

Commas can also be used to separate different selectors:


h1, h2, .special {
  margin-top: 20px;
}

This applies margin-top: 20px to any element with class special.

The "cascade" and specificity of CSS rule application

There is a reason CSS stands for Cascading Style Sheets: an element may occasionally be subject to more than one rule. CSS uses a system of specificity to determine which rule "wins."

Particulars

  • ID selectors (like #myId) are more specific than class selectors (like .myClass).
  • Class selectors are more specific than type selectors (like p, div).
  • Browsers assign weights to selectors to determine which rule applies.

Cascade and source order

The rule that appears later in the CSS (or "closer" to the element) wins if two rules have the same level of specificity. The "cascading" behavior includes this..

Relevance / !important

By appending !important, you can make a declaration take precedence over others! crucial to it (although this is frequently discouraged as it can make CSS more difficult to maintain).

Inheritance

Child elements inherit certain CSS properties from their parent. If you set color: green; on <body>, for instance, many text elements inside will take on that color unless they are overridden. However, not every property is inherited.

History of JavaScript

In 1995, Brendan Eich developed JavaScript while being at Netscape Communications Corporation. At first it was called Mocha then LiveScript and in December 1995 it became JavaScript. Part of the reason for the name change was to capitalize on the then-current popularity of the Java programming language. JavaScript lets developers carry out tasks like form validation, basic animations and other user interactions right within the browser. It was created to make web pages more interactive.

The ECMAScript standard was created in 1996 after JavaScript was submitted to ECMA International for standardization. The 1997 release of ECMAScript 1 was followed by the 1998 release of ECMAScript 2, which primarily contained minor editorial revisions. A major turning point was reached in 1999 with the release of ECMAScript 3, which included features like regular expressions, enhanced string handling, and try/catch exception handling. Dynamic HTML (DHTML), which combined HTML, CSS, and JavaScript to enable more dynamic and animated web pages, appeared around the same time.

Modern and responsive web applications were made possible by the 2005 release of AJAX (Asynchronous JavaScript and XML), which enabled web pages to update content dynamically without requiring a complete page reload. The release of ECMAScript 5 (ES5) in 2009 brought getters/setters, JSON support, and strict mode. After ECMAScript 6 (ES6/ES2015) was released in 2015, JavaScript's functionality and readability were significantly increased by adding significant features like arrow functions, template literals, classes, and modules.

Every year since 2016, ECMAScript has been updated with new features to improve the language. JavaScript is now even more flexible for creating web applications, server-side applications, and even mobile apps thanks to innovations like async/await, shared memory, and sophisticated built-in objects. JavaScript is still one of the most popular programming languages in use today, supporting interactive websites and serving as the foundation for a vast number of contemporary applications.

Pre and Post Incrementation in JavaScript

In JavaScript, incrementation is when a value is increased by 1 wether it be Pre incrementation or post incrementation, but there is a difference between them:

During pre-incrementation (++x), the variable is incremented before its value is used in the expression.

let a = 5;
let b = ++a; // a becomes 6, then b is assigned the new value of a (6)
console.log(a); // Output: 6
console.log(b); // Output: 6

During pos-incrementation (x++), the variable's current value is used in the expression before it is incremented and the expression evaluates to the original, un-incremented value of the variable.

let x = 5;
let y = x++; // y is assigned the current value of x (5), then x is incremented to 6
console.log(x); // Output: 6
console.log(y); // Output: 5
How do websites keep our passwords secure?

Websites ensure password safety by never storing them in plain text format. When a user creates a password, the site generates a unique random "salt" just for that user's account, and combines it with the password. The salted password is then processed using a one-way cryptographic hash function - systems that apply algorithms (that are often slow and contain strong computation power) such as bcrypt, Argon2, or PBKDF2 - that produces a unique string of fixed length. The site only stores the salt and the hash - the actual password is never stored at all. When a user logs into their account, the site takes the password the user provided, uses the same function, and then compares the hash generated by the verification process with the hash stored in the site - if they match, the user is granted access to their account. The protections from this system, lend themselves from the fact that the hash cannot be easily reverted, the unique salted string makes it exceptionally difficult for attackers to use precomputed "rainbow tables", and the slow hashing algorithm helps foil brute-force attacks due to being costly in time. Weak and/or reused passwords are still inherently weak and thus not secure; further, if attackers gain access to a web application/website and have access to the database they still have the ability to guess passwords one at a time, but again it is much more difficult than if the password was simply stored in plain text
